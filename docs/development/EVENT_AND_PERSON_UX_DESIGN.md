# 事件与人物阅读体验改进设计

> **文档目的**：定义事件数量控制、事件等级筛选、正文事件标记、人物关联事件展示、人物 AI 问答等功能的完整设计。

**版本**：v1.0  
**最后更新**：2026-02-27

---

## 1. 概述

### 1.1 背景与问题

| 问题 | 现状 | 目标 |
|------|------|------|
| 事件过多 | 项羽本纪 59 个、高祖本纪 99 个，列表冗长 | 合理控制数量，聚焦重要事件 |
| 事件混在一起 | 无等级区分，L1/L2/L3 混排 | 支持按等级筛选，默认只展示高等级 |
| 正文与事件脱节 | 需在右侧列表查找对应事件 | 正文中标记事件入口，可快速定位 |
| 人物信息单薄 | 点击人物仅切换 tab，无深化 | 展示关联事件，支持 AI 追问（性格、影响等） |

### 1.2 功能目标

1. **事件数量控制**：提取与展示阶段设置适当上限
2. **事件等级筛选**：使用 SegmentedControl，支持 L1 / L1+L2 / L1+L2+L3 / 全部
3. **正文事件标记**：在段落合适位置插入事件快捷入口，点击可跳转到事件或弹窗
4. **人物关联事件**：点击人物可查看其参与的本章事件
5. **人物 AI 问答**：在人物详情中支持提问（性格、历史影响、与某人关系等）

## 2. 事件相关设计

### 2.1 事件数量上限

#### 2.1.1 数据提取阶段（llmExtractor）

**策略**：在第二阶段（详情提取）对每个章节的事件总数设上限，超出部分保留概览但不提取详情。

| 等级 | 上限建议 | 说明 |
|------|----------|------|
| L1 | 无上限 | 必须全部提取 |
| L2 | 每章节 ≤25 | 超出时按时间优先保留 |
| L3 | 每章节 ≤30 | 超出时按时间优先保留 |
| **总计** | **≤80** | 单章节事件详情总数，超出则截断 |

**实现要点**：
- 在 `extractEventsFromChapter` 中，`eventsToExtract` 构建后按 `[...l1Events, ...l2Events.slice(0, 25), ...l3Events.slice(0, 30)]` 再取前 80 个
- 被截断的事件可写入 `truncatedEvents` 供后台统计，不落库

**配置化**（可选）：
- 在 `prompts.ts` 或环境变量中定义 `EVENT_L2_MAX`、`EVENT_L3_MAX`、`EVENT_TOTAL_MAX`，便于调优

#### 2.1.2 展示阶段（API / 前端）

**策略**：API 支持 `importance` 筛选；前端默认请求时可选等级，减少初次加载数据量。

- 后端 `/api/events/by-chapter/:chapterId` 增加 query 参数 `importance`：`L1` | `L1,L2` | `L1,L2,L3` | `all`
- 默认 `importance=L1`，只返回 L1 事件
- 前端切换 SegmentedControl 时改变请求参数

**注意**：已入库的事件不会因上限而删除，展示阶段的筛选只是过滤。提取阶段的上限主要影响新章节的数据质量。

### 2.2 事件等级筛选（SegmentedControl）

#### 2.2.1 UI 设计

在 **EventTimeline** 组件顶部（"共 X 个事件" 上方或同一行），增加 SegmentedControl：

```
[ L1 ] [ L1+L2 ] [ L1+L2+L3 ] [ 全部 ]
```

- **L1**：仅最高等级（改朝换代、主角生死、决定性战役等）
- **L1+L2**：L1 + 重要政治/军事事件
- **L1+L2+L3**：L1 + L2 + 关键转折点、重要人物生死等
- **全部**：不按 importance 过滤，显示本章所有已发布事件

**默认选项**：`L1` 或 `L1+L2`（可配置，推荐 `L1+L2` 以平衡信息量与可读性）

#### 2.2.2 数据流

```
EventTimeline
  ├── importanceFilter: 'L1' | 'L1,L2' | 'L1,L2,L3' | 'all'
  ├── getEventsByChapter(chapterId, { importance: importanceFilter })
  └── 根据筛选结果渲染 Timeline
```

#### 2.2.3 API 变更

**GET `/api/public/events/by-chapter/:chapterId`**

新增 query 参数：

| 参数 | 类型 | 默认 | 说明 |
|------|------|------|------|
| importance | string | `L1,L2` | `L1` / `L1,L2` / `L1,L2,L3` / `all` |

后端逻辑：
- `L1`：`where.importance = 'L1'`
- `L1,L2`：`where.importance IN ['L1','L2']`
- `L1,L2,L3`：`where.importance IN ['L1','L2','L3']`
- `all`：不添加 importance 条件

排序保持不变：按段落顺序 + 时间。

### 2.3 正文段落中的事件标记

#### 2.3.1 标记位置

事件与段落的关联已存在于 `Event.relatedParagraphs`。需要确定的是：**在段落内具体哪个位置显示事件入口**。

**方案 A：段落末尾追加标签（推荐）**

在每个段落的末尾（或段落块下方）显示与该段落相关的事件标签，例如：

```
（正文内容……）

[ 鸿门宴 ] [ 项羽分封诸侯 ]
```

- 一个段落可能对应多个事件，用多个标签
- 标签样式：小徽章、可点击，颜色可用 eventType 区分（战役=红，政治=橙等）

**方案 B：段内首次提及处插入**

在事件名称首次出现的文本位置插入可点击标记。例如原文有「鸿门宴」三个字，在「鸿门宴」下方或后面加一个小图标（如 `📍` 或 `📌`），点击可定位事件。

- 需要知道事件名在段落中的字符位置
- 可通过 LLM 或简单字符串匹配获取，实现成本较高
- 优势：与阅读动线更贴合

**建议**：先用 **方案 A**（段落末尾标签），实现简单、不侵入正文排版；后续若有需求可增加方案 B。

#### 2.3.2 点击行为

| 方案 | 行为 | 优缺点 |
|------|------|--------|
| **滚动到列表** | 右侧事件列表滚动到该事件，并高亮 | 保持当前布局，适合想对照列表阅读的用户 |
| **弹窗展示** | 弹出 Modal/Drawer 显示事件详情（名称、时间、摘要、参与者、影响等） | 聚焦单一事件，不依赖右侧面板是否展开 |
| **混合** | 若右侧面板展开且为事件 tab → 滚动到列表；否则 → 弹窗 | 兼顾两种场景 |

**推荐**：**混合方案**
- 若 `showInfoPanel && activeTab === 'events'`：滚动到事件列表并高亮该项
- 否则：打开事件详情弹窗（可包含「在时间轴中查看」按钮，切换到事件 tab 并滚动）

#### 2.3.3 数据结构与实现要点

- 段落对应事件列表：由 `paragraphEventsMap`（`Map<paragraphId, Event[]>`）提供，当前 ReadingPage 已构建
- 在 `renderParagraphContent` 或每个段落的容器底部，根据 `paragraphEventsMap.get(paragraph.id)` 渲染事件标签
- 标签点击：调用 `onEventClick(event)` 或新的 `onEventLabelClick(event, mode: 'scroll' | 'modal')`，由父组件决定行为

## 3. 人物相关设计

### 3.1 人物关联事件展示

#### 3.1.1 入口

- **PersonList** 中点击人物卡片 → 当前行为：`onPersonClick(person)`，切换 tab 并选中人物
- **改进**：点击后打开 **人物详情 Drawer/Modal**，包含：
  1. 基本信息（姓名、别名、阵营、角色、生卒、传记摘要）
  2. **本章关联事件**：该人物在本章节中参与的事件列表（可折叠）

#### 3.1.2 事件列表来源

- 已有 API：`GET /api/public/persons/:id`，query `chapterId` 时返回 `chapterEvents`
- 若无 `chapterId`，可调用 `GET /api/public/persons/:id/events` 获取全书范围的事件，再按章节分组或仅展示本章

**推荐**：阅读页始终带 `chapterId`，调用 `GET /api/public/persons/:id?chapterId=xxx` 获取 `chapterEvents`。

#### 3.1.3 UI 布局

人物详情 Drawer 结构示例：

```
┌─────────────────────────────────────┐
│ 刘邦                      [汉] 君主   │
│ 又称：沛公、汉高祖                    │
│ 前256年—前195年                     │
├─────────────────────────────────────┤
│ 传记摘要（可展开）                    │
├─────────────────────────────────────┤
│ 本章关联事件 (12)          [展开/收起] │
│   • 鸿门宴 (-206年) [查看]            │
│   • 垓下之战 (-202年) [查看]         │
│   …                                 │
├─────────────────────────────────────┤
│ 向 AI 提问                           │
│ [输入框] [发送]                       │
└─────────────────────────────────────┘
```

- 事件项可点击，行为与「正文事件标记」一致：滚动到事件列表或弹窗展示事件详情
- 可在 Drawer 内嵌「在时间轴中查看」链接，关闭 Drawer 并切换到事件 tab、滚动到对应事件

### 3.2 人物 AI 问答

#### 3.2.1 能力范围

用户可在人物详情中提问，例如：
- 该人物的性格特点
- 历史影响与评价
- 与另一人的关系（如「刘邦和项羽的关系如何」）
- 某次事件中的动机（需结合事件上下文）

#### 3.2.2 上下文构建

| 优先级 | 内容 | 字数/Tokens 建议 |
|--------|------|------------------|
| 1 | 人物基本信息与传记 | biography 全量，约 200–500 字 |
| 2 | 人物参与的本章事件摘要 | 每事件 1 句，约 50 字/事件，最多 10 个 |
| 3 | 当前章节标题、书名 | 约 20 字 |
| 4 | 对话历史 | 最近 2–3 轮 |
| 5 | 用户问题 | - |

总上下文建议控制在 3000–4000 tokens，避免超长输入。

#### 3.2.3 API 设计

**POST `/api/public/chat/person-ask`**（公开接口，无需登录）

请求体：
```json
{
  "personId": "person_xxx",
  "chapterId": "chapter_yyy",
  "question": "刘邦的性格特点是什么？",
  "conversationId": "conv_zzz"
}
```

- `conversationId` 可选，用于多轮对话
- 服务端根据 `personId`、`chapterId` 拉取人物信息与本章事件，构建 prompt，调用 LLM，返回回答

响应：
- 流式：`data: {"type":"token","content":"..."}` 等（与 specs 中 `/api/chat/ask` 类似）
- 或非流式：`{ "answer": "...", "conversationId": "..." }`

**安全与限流**：
- 公开接口需做 rate limit（如每 IP 每分钟 10 次）
- 可选：要求简单 captcha 或登录后使用

#### 3.2.4 前端界面

在人物详情 Drawer 底部：
- 输入框 + 发送按钮
- 消息列表：用户问题 + AI 回答（支持 markdown 渲染）
- 可选：预设问题按钮，如「性格特点」「历史影响」「与项羽的关系」

## 5. 附录

### 5.1 事件 importance 等级定义（与现有 spec 一致）

| 等级 | 说明 |
|------|------|
| L1 | 改朝换代、主角生死、决定性战役、关键决策 |
| L2 | 重要政治事件、重大军事行动、影响局势的事件 |
| L3 | 重要人物生死（非主角）、关键转折点、影响主要人物命运 |
| L4 | 一般战役、次要政治事件 |
| L5 | 不提取 |

### 5.2 与现有组件的衔接

- **EventTimeline**：增加 `importanceFilter` state、SegmentedControl、向 `getEventsByChapter` 传入 `importance`
- **ReadingPage**：增加 `paragraphEventsMap` 到段落渲染、`onEventLabelClick` 回调
- **PersonList**：点击人物时打开 Drawer 而非仅 `onPersonClick`；Drawer 内展示 `chapterEvents` 与 AI 问答
- **api.ts**：`getEventsByChapter` 增加可选参数 `importance`；新增 `getPersonWithChapterEvents`（或复用现有 `getPerson` + `chapterId`）

### 5.3 配置建议汇总

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 默认事件等级筛选 | L1+L2 | 平衡信息量与可读性 |
| 提取阶段 L2 上限 | 25 | 可调 |
| 提取阶段 L3 上限 | 30 | 可调 |
| 提取阶段总事件上限 | 80 | 可调 |
| AI 问答上下文 max tokens | 4000 | 可调 |

### 5.4 前端类型变更

- **Event** 接口需增加 `importance?: string` 字段，用于筛选及可选展示（如等级徽章）
