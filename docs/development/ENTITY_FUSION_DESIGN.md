# 实体聚合展示系统设计 (KISS 原则)

## 1. 背景

随着多章节内容的不断导入，同一个实体（人物、地点等）会在不同章节中被多次提及。我们需要一个简单、高效且能快速实施的方案来处理这些信息，并以合理的方式展示给用户。

## 2. 核心原则：KISS (Keep It Simple, Stupid)

我们采用**“写入时简化，读取时聚合”**的核心策略。

1.  **数据只追加 (Append-Only)**：从 LLM 提取的任何实体数据，都作为一条新记录直接存入数据库。
2.  **不修改、不融合 (No Modify, No Merge)**：在数据写入阶段，不进行任何去重、合并或更新操作。每一条记录都与其原始章节牢固绑定。
3.  **保证数据原始性**：数据库中的每一条记录都 100% 反映了其在某一章节的原始提取信息，绝无“错误合并”造成的数据污染风险。

## 3. 架构设计

### 3.1 写入流程 (Write Flow)

流程极其简单：

`LLM 提取 -> 直接存入数据库`

*   `Person`, `Place`, `Event` 等模型不需要复杂的 `canonicalId` 或 `mergeHistory` 字段。
*   `approve` 一个 `ReviewItem` 的后端逻辑就是简单地 `prisma.create()`。

### 3.2 读取与展示流程 (Read Flow)

这是本方案的核心，通过巧妙的 UX 设计来平衡用户体验和实现复杂度。

当用户查看一个实体时（例如人物“刘邦”）：

1.  **即时显示 (Zero Latency)**
    *   前端立即、无延迟地从当前上下文（如 `ReviewItem`）中获取信息，并展示该实体在**当前章节**的详细信息（如简介、阵营等）。
    *   这是主视图，保证了页面的快速响应。

2.  **异步聚合 (Asynchronous Aggregation)**
    *   在页面的特定区域（例如下方或侧边栏），会显示一个“加载中...”的提示。
    *   前端异步调用一个专门的聚合 API，例如 `GET /api/person/aggregate?name=刘邦`。
    *   这个 API 的职责是：
        a.  查询数据库中所有 `name` 为“刘邦”的记录。
        b.  将这些记录的关键信息（如不同章节的简介、参与的事件等）打包。
        c.  构建一个 Prompt，调用 LLM API，要求其“综合以下信息，生成一份关于‘刘邦’的全面总结”。
        d.  将 LLM 返回的聚合信息展示在前端。

## 4. 优缺点分析

### 优点

*   **极低的开发复杂度**：后端逻辑大大简化，可以快速实现核心功能。
*   **数据绝对可靠**：无数据污染风险，可随时追溯任何信息的来源。
*   **极高的灵活性**：未来可以随时更换更好的 LLM 模型或调整聚合逻辑（Prompt），而无需对历史数据进行昂贵的迁移和清洗。

### 权衡与优化

*   **读取延迟**：聚合信息的获取有数秒延迟。通过**异步加载**和**优先展示即时信息**的设计，将此延迟对用户体验的影响降到最低。
*   **API 成本**：每次聚合都会调用 LLM。这是一个运营成本和功能收益之间的权衡。
*   **未来优化**：可以轻松地为聚合 API 的返回结果增加**缓存 (Caching)** 机制。例如，对“刘邦”的聚合结果可以缓存 24 小时，从而大大降低后续请求的延迟和成本。

## 5. 结论

本方案以最务实、最高效的方式满足了当前阶段的需求。它将复杂性从“写入”端完全转移到了“读取”端，并通过异步加载和延迟展示的 UX 策略，优雅地解决了由此带来的体验问题，是当前阶段的最佳实践。
